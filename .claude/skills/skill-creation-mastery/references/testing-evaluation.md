# テスト・評価手法

スキルの品質を客観的に測定・改善するためのテストと評価の手法。

## テストの3つの観点

| 観点 | 質問 | テスト手法 |
|------|------|-----------|
| Activation | スキルが正しくトリガーされるか？ | Description評価 |
| Execution | スキルが正しい出力を生成するか？ | 実行テスト |
| Quality | 出力の品質は十分か？ | ベンチマーク / A/Bテスト |

## 1. Activation テスト（トリガー評価）

スキルのdescriptionが適切にマッチングされるかを評価。

### テストプロンプト集の作成

```markdown
## Positive（トリガーされるべき）
- 「GraphRAGを構築したい」
- 「知識グラフをPostgreSQLで作りたい」
- 「RAGに関係性を持たせたい」
- 「エンティティ間の関係を管理するRAGが欲しい」
- "I want to build a knowledge graph"

## Negative（トリガーされるべきでない）
- 「PostgreSQLのインデックスを最適化したい」
- 「一般的なRAGパイプラインを作りたい」
- 「SQLクエリのパフォーマンスを改善したい」
```

### 評価指標

| 指標 | 計算 | 目標値 |
|------|------|--------|
| Precision | TP / (TP + FP) | > 0.9 |
| Recall | TP / (TP + FN) | > 0.8 |
| F1 Score | 2 × P × R / (P + R) | > 0.85 |

- **TP**: 正しくトリガーされた
- **FP**: 誤ってトリガーされた（False Positive）
- **FN**: トリガーされるべきだったが、されなかった（False Negative）

### Description最適化ループ

1. テストセットを60% training / 40% test に分割
2. 現在のdescriptionで各プロンプトを3回テスト
3. 失敗ケースを分析し、description改善版を提案
4. trainingセットで評価 → testセットで検証
5. 最大5回反復
6. testスコアが最高のdescriptionを採用（過学習防止）

## 2. Execution テスト

スキルが正しく動作し、期待通りの出力を生成するかを評価。

### テストケース設計

```markdown
## テストケース1: 基本的な小説キャラクター管理
入力: 「小説のキャラクター関係を管理したい。時系列で関係が変化する」
期待出力:
- sql/schema.sql が生成される
- Entity型に Character, Location が含まれる
- 時系列サポート（events テーブル）が含まれる
- sql/queries/ にlocal_search.sql, global_search.sql が存在

## テストケース2: 技術文書ナレッジベース
入力: 「技術文書の依存関係を管理したい」
期待出力:
- Entity型に Technology, API, Component が含まれる
- Edge型に depends_on, implements が含まれる
```

### チェック項目テンプレート

```markdown
- [ ] 期待されるファイルがすべて生成されたか
- [ ] ファイルの構文が正しいか（SQL構文、JSON構文）
- [ ] ユーザー要件が反映されているか
- [ ] リファレンスのベストプラクティスに従っているか
- [ ] エラーなく完了したか
```

## 3. ベンチマーク

標準化された評価基準でスキルのパフォーマンスを追跡。

### 測定指標

| 指標 | 説明 | 測定方法 |
|------|------|----------|
| Pass Rate | テストケースの合格率 | 合格数 / 全テスト数 |
| Elapsed Time | 実行時間 | 開始〜完了の時間 |
| Token Usage | トークン消費量 | 入出力トークンの合計 |
| File Count | 生成ファイル数 | 期待ファイル数との比較 |

### ベンチマーク実行手順

1. クリーンな環境（新規セッション）で実行
2. 各テストケースを独立して実行（クロスコンタミネーション防止）
3. 結果を記録・比較

## 4. A/Bテスト

2つのスキルバージョンを比較評価。

### 実施手順

1. **バージョンA（現行）** と **バージョンB（改善版）** を用意
2. 同じテストプロンプト集で両方を実行
3. 比較エージェントが出力を**ブラインド評価**（どちらのバージョンか知らない）
4. 評価基準:
   - 正確性（要件の反映度）
   - 完全性（必要なファイルの網羅性）
   - 品質（コードの品質、ドキュメントの明瞭さ）

### 比較レポートテンプレート

```markdown
## A/Bテスト結果

| テストケース | Version A | Version B | 勝者 |
|-------------|-----------|-----------|------|
| Case 1 | 7/10 | 9/10 | B |
| Case 2 | 8/10 | 8/10 | 引分 |
| Case 3 | 6/10 | 9/10 | B |
| **合計** | **21/30** | **26/30** | **B** |

### 改善点
- Version Bはdescriptionの改善により、Case 1, 3でのトリガー精度が向上
- 実行時間: A=45s, B=38s（15%改善）
```

## 5. 回帰テスト

スキル変更時に既存機能が壊れていないことを確認。

### 実施タイミング

- descriptionの変更後
- 実行フローの変更後
- リファレンスファイルの更新後
- テンプレートの変更後

### 手順

1. 変更前のテスト結果をベースラインとして保存
2. 変更後に全テストケースを再実行
3. ベースラインとの差分を確認
4. 新たな失敗がないことを確認

## スキル品質チェックリスト

### Activation品質

- [ ] Positiveプロンプト10+件でトリガー率 > 80%
- [ ] Negativeプロンプト5+件で誤トリガー率 < 10%
- [ ] 日英両方のプロンプトでテスト済み

### Execution品質

- [ ] 基本テストケース3+件で合格
- [ ] エッジケース2+件で合格
- [ ] エラー時の回復が適切

### 構造品質

- [ ] SKILL.md 500行以内
- [ ] descriptionに WHAT + WHEN が含まれている
- [ ] リファレンスリンクがすべて有効
- [ ] frontmatterの必須フィールドが正しい
