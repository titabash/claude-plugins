# スキル合成・チェイニング

複数のスキルを組み合わせて高度なワークフローを構築するパターン。

## パターン1: メタスキル（スキルを生成するスキル）

スキルの実行結果として、別のスキルを生成・登録するパターン。

### 実例: UI Design Guideline Creator

```
入力: ユーザーの要件（プロジェクト名、カラー等）
実行: 要件に基づいてデザインガイドラインを設計
出力: .claude/skills/design-guideline/ にスキルを生成
```

生成されたスキルは以後のセッションで自動的に参照される。

### メタスキル設計のポイント

1. **生成先の明示**: 出力されるスキルのパスをSKILL.mdに明記
2. **既存チェック**: 同名スキルが存在する場合の上書き確認
3. **生成スキルの品質**: テンプレートを使い、生成されるSKILL.mdも500行以内に
4. **自己参照回避**: メタスキルが自身を上書きしないよう注意

```markdown
## Output Location
生成スキルの配置先:
.claude/skills/design-guideline/
├── SKILL.md
└── references/
    ├── colors.md
    ├── typography.md
    └── spacing.md
```

## パターン2: SharedWorkflowContext

複数のスキルが共有状態オブジェクトを介して連携するパターン。

### 概念

```
Skill A → WorkflowContext → Skill B → WorkflowContext → Skill C
```

各スキルは:
1. WorkflowContextから必要なフィールドを読み取る
2. 自身のタスクを実行
3. 結果をWorkflowContextに追記

### 実装例

```json
// .workflow-context.json
{
  "project": {
    "name": "my-app",
    "platform": "web"
  },
  "design": {
    "colors": { /* Skill Aが生成 */ },
    "typography": { /* Skill Bが生成 */ }
  },
  "components": {
    /* Skill Cが生成 */
  }
}
```

### 設計原則

1. **単一状態ファイル**: 複数ファイルではなく1つのJSONに集約
2. **スキーマの合意**: 各スキルが読み書きするフィールドを明確に定義
3. **追記のみ**: 他スキルのフィールドは読み取りのみ、上書きしない
4. **Claudeがオーケストレーター**: スキル間の呼び出し順序はClaudeが判断

## パターン3: パイプラインチェイニング

スキルの出力が次のスキルの入力になるシーケンシャルパターン。

### 例: RED-GREEN-REFACTOR

```
/analyze (RED) → テスト仕様生成
    ↓
/implement (GREEN) → 仕様を満たすコード実装
    ↓
/refactor (REFACTOR) → コード品質改善
```

各コマンドは独立したスキルだが、前のスキルの出力ファイルを入力として使用。

### 設計のポイント

1. **出力の標準化**: 各スキルの出力フォーマットを統一
2. **中間ファイルの場所**: 共通ディレクトリ（例: `.workflow/`）に配置
3. **冪等性**: 各スキルは何度実行しても同じ結果

## パターン4: 補完スキル

メインスキルの出力を別スキルが補完するパターン。

### 例

```
main-skill: GraphRAGスキーマ生成
    ↓
supplementary-skill: 生成されたスキーマのバリデーション
    ↓
supplementary-skill: パフォーマンステスト生成
```

## 分割 vs 統合の判断

### スキルを分割すべきケース

- 各部分が独立して使用される可能性がある
- 異なるユーザー（設計者 vs 開発者）が使用する
- 実行時間が長く、部分的な再実行が必要
- 異なるallowed-toolsが必要

### 統合すべきケース

- 常に一緒に実行される
- 共有状態が複雑で、分離するとオーバーヘッドが大きい
- フェーズ数が5以下で十分管理できる
- 同じallowed-toolsで動作する

## 合成スキルのアンチパターン

### 1. 過度な分割

```
# 悪い例: 3行の処理を別スキルにする
skill-a: ファイル読み込み
skill-b: 変換
skill-c: 書き込み
```

### 2. 暗黙の依存関係

```
# 悪い例: Skill Bが暗黙的にSkill Aの出力に依存
# ドキュメントに記載されていない
```

### 3. 循環参照

```
# 悪い例
Skill A → Skill B → Skill A（無限ループ）
```

## チェックリスト

- [ ] 各スキルの入出力が明確に定義されているか
- [ ] 依存関係がドキュメント化されているか
- [ ] 循環参照がないか
- [ ] SharedWorkflowContextのスキーマが合意されているか
- [ ] 各スキルが独立してテスト可能か
